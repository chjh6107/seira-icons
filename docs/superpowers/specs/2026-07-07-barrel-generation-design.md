## 1. 배경과 목표

`add`는 요청 아이콘 `.tsx`를 각각 복사하지만 배럴이 없어 소비자는 아이콘마다 개별 import를 써야 한다:

```tsx
import Heart from '@/components/icons/heart';
import Star from '@/components/icons/star';
```

사용자 요청(2026-07-07): **단일 import 창구**.

```tsx
import { Heart, Star } from '@/components/icons';
```

`add` 시 대상 폴더 `index.ts`(배럴)를 자동 유지한다. **additive** 기능이다 — 배럴로도, 개별 파일로도 import 가능(deep import 유지).

### 목표
- `add` 성공 시(≥1 인식된 아이콘) 대상 폴더 `index.ts`를 폴더 내용에서 **결정적으로 재생성**.
- 정식 PascalCase named re-export + `IconProps` 타입 re-export.
- 사용자 소유 `index.ts`를 **어떤 경우에도 조용히 파괴하지 않음**.
- 기본 ON, `--no-barrel` 옵트아웃(지속 저장).

### 비목표
- foreign(비-아이콘) `.tsx` re-export, watch 모드, 파일 삭제 자동 감지(재실행-add가 폴더 스캔으로 정리).
- 배럴 병합/부분 갱신(전체 재생성).
- 사용자 index.ts 자동 채택 플래그(`--adopt-barrel` 등) — 채택은 파일 삭제 후 재실행으로(문서화). YAGNI.

## 3. 생성물 형식 (canonical)

```ts
// Generated by @seira-icons/ionicons — do not edit.
export type { IconProps } from './icon.types';
export { default as Heart } from './heart';
export { default as HeartOutline } from './heart-outline';
export { default as Star } from './star';
```

- **정확 바이트**: LF 개행, 파일 끝 단일 `\n`. (재실행 시 동일 바이트 → no-op)
- 1행: **마커 헤더**. 완전 문자열 `// Generated by @seira-icons/ionicons — do not edit.`(em-dash 포함). **매칭은 ASCII 토큰 `// Generated by @seira-icons/ionicons`의 startsWith로**(em-dash 이후는 비교하지 않음).
- 타입 줄은 `icon.types.ts`가 폴더에 있을 때만.
- 아이콘 줄: 폴더의 `*.tsx`(자연히 `index.ts`·`icon.types.ts` 제외) 중 **export명 맵에 있는 것만**, kebab 파일명 코드유닛 정렬.

## 4. export명 소스 파싱 (fail-fast)

패키지 `icons/index.ts`는 1357개의 `export { default as <Pascal> } from './<kebab>';`(형식 균일·UTF-8 LF, 리뷰 실측 1357/1357 매칭). 1회 파싱해 `kebab→Pascal` 맵 생성(메모이즈).

- 정규식(줄 단위): `/^export \{ default as (\w+) \} from '\.\/([\w-]+)';$/`.
- **가드**: 파싱 후 `map.size`가 붕괴(예: `< 1000`)면 **throw**("export map parse collapsed — package icons/index.ts format changed"). 조용한 빈-배럴 방지.
- 맵은 이름 소스이자 **"이게 진짜 아이콘인가" 게이트**: 대상 폴더 파일명이 맵 key일 때만 re-export(맵 **value** 방출). 임의 파일명이 배럴로 새는 것 차단.
- 이 파일은 `files:["...","icons",...]`로 동봉되어 런타임 존재.

## 5. 덮어쓰기 안전 (마커 인지, ensureFile에 통합)

핵심: 마커 판정을 **별도 read로 하지 않고** 기존 `ensureFile`의 단일 `lstat`+read 안에서 수행 → 심링크 미보호 read/TOCTOU 제거.

- `src/cli/fs-safe.js`의 `ensureFile`에 선택 인자 `allowOverwrite?: (existing: string) => boolean` 추가. 파일이 존재하고 내용이 다를 때, `allowOverwrite`가 있으면 그 결과로, 없으면 기존 `overwrite` 불리언으로 덮어쓸지 결정. `lstat` 심링크 거부 + `O_EXCL|O_NOFOLLOW`/`O_TRUNC|O_NOFOLLOW`는 불변.
- 배럴은 `ensureFile(index.ts, content, { compare: normalizeEol, allowOverwrite: isManaged })`로 호출:
  - 없음 → 생성(`written`)
  - 마커 있음(우리 것) → 갱신(`written`, 아이콘 추가 반영)
  - 내용 동일(개행 정규화 후) → `skipped-identical`(no-op)
  - 마커 없음(사용자 소유) → `conflict` = **비클로버** → 경고
- **`--overwrite`는 배럴에 전달하지 않음**(디컨플레이션): 아이콘 refresh용 `--overwrite`가 사용자 index.ts를 파괴하지 않는다. managed 배럴은 어차피 항상 재생성되므로 `--overwrite` 불필요.
- `isManaged(content)`: 선행 BOM(`﻿`) 제거 + 첫 줄 trailing `\r`/공백 제거 후 ASCII 토큰 startsWith. 단일 `MARKER`/`MARKER_TOKEN` 상수를 `buildBarrel`과 공유 → 자기 출력이 다음 실행에 반드시 재인식(round-trip).
- 사용자 채택: 자기 index.ts를 지우고 `add` 재실행(문서화).

## 6. 아키텍처

```
src/cli/barrel.js
  - MARKER: string                            # 1행 전체(생성용)
  - MARKER_TOKEN: string                       # 매칭용 ASCII 토큰
  - exportNameMap(): Map<kebab,Pascal>         # 패키지 index.ts 파싱+가드, 메모이즈
  - buildBarrel(iconBasenames, {hasTypes}): string   # 순수: 정렬+마커+타입줄+export줄, canonical
  - isManaged(existing): boolean               # 마커 정규화 판정
  - ensureBarrel(realDir, {}): {status, count} # 폴더 스캔 → 맵 조회 → ensureFile(allowOverwrite:isManaged)
```

- `src/cli/fs-safe.js`: `ensureFile`에 `allowOverwrite` predicate 추가(하위호환).
- `src/cli/add.js`: 아이콘 쓰기 후 `barrel !== false && ok.length>0`이면 `ensureBarrel(realDir)` 호출, 리포트에 `barrel:{status,count}` 추가.
- `bin/cli.js`: `--no-barrel`(boolean) 파싱 → `barrel` 결정(설정 `barrel:false`도 반영), `addIcons`에 전달. `--no-barrel` 사용 시 `writeConfig`에 `barrel:false` 지속. HELP에 `--no-barrel` 추가. 배럴 상태를 `+/=/!` 심볼로 출력.
- `src/cli/config.js`: 스키마에 선택 `barrel?: boolean` 추가(타입가드 읽기, deep-merge 없음).
- 순수 `buildBarrel`/`isManaged`와 I/O `ensureBarrel` 분리 → fs 없이 단위 테스트.

### 6.1 모듈 계약
- `exportNameMap() -> Map<string,string>` (가드 실패 시 throw)
- `buildBarrel(iconBasenames: string[], { hasTypes: boolean }) -> string` (내부 정렬, canonical)
- `isManaged(existing: string) -> boolean`
- `ensureBarrel(realDir: string, opts?: {}) -> { status: 'written'|'skipped-identical'|'skipped-unmanaged', count: number }`
- `ensureFile(path, content, { overwrite?, compare?, allowOverwrite? })` — `allowOverwrite(existing)` 우선.
- `addIcons` 리포트에 `barrel?: { status, count }`(미실행 시 생략).
- 전제조건: `ensureBarrel`의 `realDir`는 호출자가 `assertInsideProject`로 confine한 경로(문서화).

## 7. 데이터 흐름 (`add star heart`)
1. 아이콘/공유파일 기존대로 복사(ok.length>0).
2. barrel 켜짐이면 `ensureBarrel(realDir)`:
   a. `readdirSync(realDir)`의 `*.tsx`(=index 제외) 수집.
   b. `exportNameMap`으로 각 파일명→export명(맵 key 아니면 제외, foreign 카운트).
   c. `hasTypes = existsSync(realDir/icon.types.ts)`.
   d. `buildBarrel(names,{hasTypes})` → `ensureFile(..., {compare:normalizeEol, allowOverwrite:isManaged})`.
3. CLI 출력: `+ index.ts (barrel, N exports)`(생성) / `= index.ts (barrel, up to date)`(동일) / `! index.ts not managed — left unchanged (pass --no-barrel to silence)`(비클로버). foreign 있으면 `i N file(s) not recognized icons — not exported`. 모두 stdout, exit 0.

## 8. 에러 처리 / 엣지
- 마커 없는 사용자 index.ts → 비클로버 + 경고(설치 실패 아님, exit 0). `--no-barrel`로 지속 침묵.
- BOM/CRLF된 우리 배럴 → 정규화로 여전히 managed 인식(무한 skip/staleness 방지).
- 개행만 다른 동일 배럴 → `compare:normalizeEol`로 `skipped-identical`(Prettier 등과의 churn 완화; 추가로 README에 `.prettierignore` 안내).
- foreign `.tsx` → 배럴 제외 + 카운트 알림.
- index.ts가 디렉토리/FIFO 등 비정상 → 배럴 단계에서 예외를 잡아 `skipped-unmanaged`+경고로 다운그레이드(아이콘은 이미 정상 복사됨, 전체 add 실패 안 함).
- `ok.length===0`(전부 unknown) → 배럴 미실행.
- 배럴 쓰기도 confinement(호출자)+no-follow(ensureFile) 적용.
- 수동 아이콘 삭제 → 다음 `add`에서 폴더 스캔으로 자동 정리(그 전까진 dangling — 비목표, README 명시).
